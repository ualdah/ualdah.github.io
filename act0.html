1. Configuración inicial

  • Instalaremos la última versión LTS de Node.js (enlace).

Comprobaremos que se han instalado correctamente Node.js y npm:

$ node -v
v22.14.1

$ npm -v
10.9.2

  • Instalaremos Ionic, incluyendo algunas dependencias relacionadas con los dispositivos móviles:

npm install -g @ionic/cli native-run cordova-res

Comprobaremos que Ionic ha sido instalado correctamente:

$ ionic info
[WARN] You are not in an Ionic project directory. Project context may be missing.

Ionic:

   Ionic CLI : 7.2.0

Utility:

   cordova-res : 0.15.4
   native-run  : 2.0.1

System:

   NodeJS : v22.14.0
   npm    : 10.9.2
   OS     : Windows 10

La advertencia nos indica que no nos encontramos en un directorio que contenga un proyecto de tipo Ionic.

El siguiente paso es crear una aplicación mediante el comando:

$ ionic start <name> <template> [options]

El primer parámetro <name> indica el nombre de la aplicación. El segundo parámetro <template> indica el nombre de la plantilla a partir de la cual se crea el proyecto. El tercer parámetro permite indicar un tipo de proyecto específico con --type, la utilización de Apache Cordova con --cordova o la utilización de Ionic Capacitor con --capacitor. Mediante el comando ionic start --list se listan los diferentes tipos de proyectos y de plantillas.

Para que funcione correctamente el script de ionic start es necesario que la ruta actual no contenga espacios ni caracteres especiales

  • Crearemos una aplicación con menú y que utilice Capacitor:

$ ionic start myAppMenu sidemenu --type=angular --capacitor

  • Más adelante veremos la diferencia entre crear el proyecto utilizando NgModules o Standalone Components. Para este primer ejemplo, seleccionaremos la opción por defecto de NgModules.

  • Desplegaremos la aplicación:

$ cd .\myAppMenu
$ ionic serve

Después de la configuración del despliegue, se mostrará la localización de la aplicación desplegada:

[INFO] Development server running!

       Local: http://localhost:8100

       Use Ctrl+C to quit this process

[INFO] Browser window opened to http://localhost:8100!

2. Aplicación de ejemplo con Ionic

En primer lugar, creamos una aplicación a partir de la plantilla tabs cuyo nombre sea myAppTabs:

$ ionic start myAppTabs tabs --type=angular --capacitor

Si nos fijamos en la estructura del proyecto, podemos ver que se han creado las siguientes carpetas:

  • .vscode: Archivos de configuración de Visual Studio Code.

  • node_modules: Contiene los paquetes de los que depende la aplicación.

  • src: Contiene el código fuente de la aplicación.

Además, se ha creado un conjunto de archivos de configuración:

  • .browserslist: Lista de navegadores a tener en cuenta para la generación del código CSS y JavaScript.

  • .editorconfig: Indica la configuración para la edición y visualización de los archivos de texto del proyecto.

  • .eslintrc.json: Permite configurar ESLint para el análisis del código y la solución rápida de problemas.

  • .gitignore: Especifica los archivos que no deben tenerse en cuenta en la gestión de versiones.

  • angular.json: Contiene la configuración de Angular CLI.

  • capacitor.config.json: Contiene la configuración de la integración de Ionic Capacitor por plataforma. Similar a config.xml para Cordova.

  • ionic.config.json: Contiene la configuración del proyecto Ionic. Su nombre, el tipo, las integraciones configuradas, etc.

  • karma.config.json: Establece la configuración para la realización de las pruebas unitarias con Karma.

  • package-lock.json: Archivo generador a partir de package.json que contiene el árbol de dependencias completo derivado del mismo.

  • package.json: Describe metadatos del proyecto y también especifica sus dependencias.

  • tsconfig.app.json: Configuración específica del directorio app para realizar la conversión de TypeScript a JavaScript.

  • tsconfig.json: Establece la configuración mediante la que el compilador de TypeScript realiza la conversión a JavaScript.

  • tsconfig.spec.json: Configuración específica para la realización de las pruebas.

2.1. Contenido de la carpeta src

Dentro de la carpeta del código fuente de la aplicación existen los siguientes directorios:

  • app: Contiene los módulos y los componentes de la aplicación, además de la especificación de la navegación entre ellos.

  • assets: Contiene los recursos utilizados en la aplicación, incluyendo también los iconos y las fuentes.

  • environments: Contiene archivos de configuración para diferentes entornos. Por defecto, se definen los entornos de desarrollo (production : false) y producción (production : true).

  • theme: Contiene los archivos Sass que permiten personalizar la apariencia de la aplicación. El archivo variables.scss describe variables con colores para diferentes temas o estilos.

Además, se han generado los siguientes archivos:

  • global.scss: Especifica los estilos que se aplican de forma global a toda la aplicación.

  • index.html: Archivo principal que sirve de punto de entrada para el despliegue de la aplicación en el componente WebView.

  • main.ts: Inicia la aplicación, realizando la carga del entorno.

  • polyfills.ts: Contiene los polyfills (fragmentos de código que proporcionan funcionalidad moderna en navegadores antiguos o que no soportan cierta funcionalidad de forma nativa) que requiere Angular.

  • test.ts: Punto de entrada para la ejecución de las pruebas unitarias haciendo. Realiza la carga de todos los archivos .spec.

  • zone-flags.ts: Se utiliza para evitar que se generen ciertos eventos o acciones asíncronas.

Dentro de la carpeta app se crea una carpeta por cada elemento del proyecto (páginas, componentes, servicios, módulos, etc.).

En las siguientes subsecciones, se describen algunos de los archivos más importantes para comprender la estructura del proyecto Ionic generado.

2.3. El componente app-root

Este componente se encuentra dentro de la carpeta app y se describe mediante los archivos:

  • app.component.html

  • app.component.scss

  • app.component.spec.ts

Comenzaremos con la descripción del archivo con extensión .ts, puesto que es el punto de partida de cada uno de los componentes.

2.4. El archivo src > app > app.module.ts

Debido a que la aplicación Ionic es de tipo Angular y basada en NgModules en lugar de Standalone Components, los componentes, proveedores de servicios y directivas, deben ser declaradas en este archivo.

Archivo src/app/app.module.ts

import { NgModule } from '@angular/core';
import { BrowserModule } from '@angular/platform-browser';
import { RouteReuseStrategy } from '@angular/router';

import { IonicModule, IonicRouteStrategy } from '@ionic/angular';

import { AppRoutingModule } from './app-routing.module';
import { AppComponent } from './app.component'; (1)

@NgModule({
  declarations: [AppComponent], (2)
  entryComponents: [],
  imports: [BrowserModule, IonicModule.forRoot(), AppRoutingModule],
  providers: [{ provide: RouteReuseStrategy, useClass: IonicRouteStrategy }],
  bootstrap: [AppComponent], (3)
})
export class AppModule {}
1 Importación del componente
2 Declaración de los componentes del módulo
3 Ejecución del componente en la carga del módulo

2.5. El archivo src > app > app-routing.module.ts

Describe el enrutamiento (navegación) de la aplicación. En la ruta por defecto o vacía (path : '') se realiza una carga del módulo declarado en tabs.module.ts.

Archivo src/app/app-routing.module.ts

import { NgModule } from '@angular/core';
import { PreloadAllModules, RouterModule, Routes } from '@angular/router';

const routes: Routes = [
  {
    path: '', (1)
    loadChildren: () => import('./tabs/tabs.module').then(m => m.TabsPageModule) (2)
  }
];
@NgModule({
  imports: [
    RouterModule.forRoot(routes, { preloadingStrategy: PreloadAllModules })
  ],
  exports: [RouterModule]
})
export class AppRoutingModule {}
1 Ruta por defecto a la aplicación
2 Carga del módulo tabs.module.ts

2.7. Las páginas tab1, tab2 y tab3

Una página (page) es un componente al que se le crea un módulo (module) y una definición de navegación (routing.module) de forma similar a la definición del componente principal de la aplicación (app).

En esta aplicación, se han creado tres pestañas muy similares.

La diferencia se encuentra en la definición del contenido de su correspondiente contenido del componente pestaña (tab1, tab2 y tab3) y en el uso que hacen del componente explore-container.

Archivo src/app/tab1/tab1.page.html

<ion-header class=".ion-no-padding" [translucent]="true">
  <ion-toolbar>
    <ion-title>
      Tab 1
    </ion-title>
  </ion-toolbar>
</ion-header>

<ion-content [fullscreen]="true">
  <ion-header collapse="condense">
    <ion-toolbar>
      <ion-title size="large">Tab 1</ion-title>
    </ion-toolbar>
  </ion-header>

  <app-explore-container name="Tab 1 page"></app-explore-container> (1)
</ion-content>

2.8. La página src > app > tabs

En la aplicación se crea también un componente de tipo página que contiene y realiza la carga de las pestañas tab1, tab2 y tab3.

Para cada pestaña, se establece un icono y una etiqueta.

3. Ejercicio: Modificación de la aplicación

Como ejercicio, se propone llevar a cabo una modificación de la aplicación creada para que se cumplan las siguientes características:

  • Las pestañas deben referirse a:

    • Home

    • List

    • Action

  • Los iconos de acceso a las pestañas deber ser diferentes a los generados en la aplicación de forma automática. Por ejemplo, una casa, una lista y una rueda de un mecanismo.

  • Los contenidos de cada pestaña, deben mostrar un nombre o título en negrita y luego un texto con una descripción

  • Los textos de las tres pestañas deben aparecer exactamente a la misma altura, tres píxeles por debajo de la mitad del componente y centrados.
